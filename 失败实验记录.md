# 红节点架构实验系统 — 发散性爆开问题报告

## 一、项目背景

当前游戏的节点系统正在进行一次**结构级升级实验**。

实验目标：

将部分节点升级为一种 **刚性结构节点（红节点架构）**，实现新的结构玩法能力：

* 节点之间可以形成**稳定结构**
* 玩家可以在编辑模式下对结构进行**塑形**
* 在动力学中结构表现为**整体受力，而不是单节点拖拽**

该实验系统：

* 默认关闭
* 通过 **调试面板开关**启用
* 不影响现有正式玩法

---

# 二、实验设计目标

## 1 基础节点统一

原系统存在多种节点类型。

实验方案：

* 所有节点默认统一为 **蓝色节点**
* 蓝色节点行为保持现有逻辑

---

## 2 红节点的定义

节点可以被染为 **红色节点**。

操作方式：

* 右键点击节点 → 染红
* 多选节点 → 右键批量染红
* Alt + 右键 → 取消红色

---

## 3 红节点连接逻辑

红节点之间：

* 自动生成 **红色连线**
* 红线表示 **刚性结构连接**

红线复用已有的：

**多根线结构固定系统（3 根杆）**

原系统：

```
1 根线
2 根线
3 根线
```

实验只使用：

```
3 根线
```

其他模式关闭。

---

## 4 红节点动力学行为

当脉冲驱动到红节点时：

旧系统：

```
探出 → 定住 → 拉过去
```

红节点实验改为：

```
对红结构整体施加推动力
```

效果类似：

**推动整个结构**

而不是拖拽单节点。

---

## 5 编辑模式塑形能力

编辑模式下：

拖拽红节点：

* 保持整体结构稳定
* 根据拖拽进行**形状重塑**

示例：

```
三角形
↓
拖动一个点
↓
等腰三角形
```

# 四、首次出现的问题

当**多选节点同时变红**时出现严重不稳定：

### 现象

节点会：

* 瞬间聚拢
* 高速旋转
* 爆飞出屏幕
* 摄像机无法捕捉

类似：

```
物理发散
```

用户描述：

> 类似“左脚踩右脚”的发散

---

# 五、第一次排查与修复

Codex 判断的原因：

### 问题1：红杆 rest 长度错误

红线的 rest length 没有使用当前形状。

而是使用：

```
旧 slot 拓扑
```

导致：

```
红化瞬间 → 强制拉回旧结构
```

修复：

```
红化时记录当前姿态为 rest
```

---

### 问题2：重复约束

红节点之间存在：

```
普通蓝边
+
红杆
```

两套约束同时生效。

修复：

```
红-红节点只保留红杆物理
普通边保留但不参与求解
```

---

### 问题3：红杆长度限制太激进

原默认：

```
64 ~ 260
```

改为：

```
0 ~ 2000
```

避免瞬间收缩。

# 八、发现的关键问题

红结构使用：

```
完全互连 (clique)
```

如果：

```
n = 6
```

则：

```
pair = 15
```

每 pair：

```
3 条杆
```

最终：

```
45 条物理边
```

这是：

```
O(n²) 约束密度
```

导致：

```
约束竞争
```

出现：

> 左脚踩右脚

---

# 九、第二次结构修改

改为：

### 稀疏刚性骨架

结构：

```
1 圈环
+
扇形支撑
```

约束数量：

```
≈ 2n - 3
```

示例：

```
6 节点

原：15 pair
新：9 pair
```

物理边：

```
27
```

### 1 约束系统仍然存在冲突

```

+

+
```

### 2 结构求解顺序问题

可能存在：

```
拖拽
→ PBD
→ 红结构
→ 蓝结构
```

循环反馈。

---

### 3 速度或动量没有正确守恒

可能出现：

```
重复叠加 impulse
```

---

### 4 三杆结构在多节点情况下不稳定

可能需要：

```
真正的 rigid body solver
```

而不是：

```
大量 distance constraints
```
