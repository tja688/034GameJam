# 猎物生成与分布系统分析：从“程序刷怪”到“生态散落”

在详细阅读了 `Docs/猎物状态机与追逃机能说明.md`、`README.md` 以及 `src/sceneProgression.js` 和 `src/sceneEnemies.js` 等核心业务逻辑后，对于你提出的 **“每次开场很多，平时路上很少，不像广阔散落的群落而像程序刷的怪”** 这个微妙的体验差，这里给出一个明确的诊断：

你的感觉完全正确，现在的代码并不是在构建一个“世界”，而是非常传统的 **“街机式补给填充加上了一个极其拖后腿的隐藏 Bug（未完全释放的产能）”**。

以下是具体的梳理以及出现这种体验的根本原因。

---

## 1. 当前生成、分布与概率的底层逻辑

在目前的架构下，猎物并没有放在一个真实的“广阔空间大地图”里，游戏的地图实际上是无限（Infinite）的，系统所有对于坐标的推演都在以“随着玩家质心相对移动”为参考基准。目前的逻辑可以分为两个阶段：

### 阶段一：开局/切关生成 (Seed Spawn) - “为什么一开始很多”
* **代码入口**：`sceneEnemies.js` -> `seedConfiguredPrey`
* **逻辑**：在刚进入当前 Stage 时（或由代码复位时），通过 `getStageInitialSpawnCount` 获取比平常更高的期望数目（有密度 Multiplier 的加算），然后强制在玩家位置周围画多个圆环（距离介于 `minRadius` 和 `maxRadius`），并在不同的角度阵列式排开产生猎物。
* **表现**：这制造了一个**“你出生在一个生物群落中间”**的静态错觉，所以第一眼印象非常丰满。

### 阶段二：日常移动时生成 (Update Spawns) - “为什么像程序刷的”
* **代码入口**：`sceneEnemies.js` -> `updateSpawns` 和 `spawnConfiguredPrey`
* **逻辑**：游戏有一个基于定时器的轮询（由 `DEMO_STAGE_DEFS` 中的 `desired` 和 `interval` 定义参数）：
  1. 系统每帧检查场上的 `alive` 数量。
  2. 如果当前这种类型的活着的怪物小于它在这一关的期望值（`alive < rule.desired`），定时器就开始走。
  3. 定时器归零时，就会随机在屏幕的**边缘**（左右或者上下的一侧）强行刷一波，由 `margin`（写死在 260，即屏幕外一点点）来控制。
* **表现**：这非常像很多街机游戏（例如捕鱼达人、雷电）。这些怪是冲着屏幕方向凭空捏造在屏幕切线上的，它们的初速度会使它们切入屏幕。这缺乏了“玩家走到一个地盘，发现一堆怪在吃草”那种探索未知边界的生态感。

---

## 2. 根本矛盾与 Bug：为什么“平时走路上很少”？

导致你产生“平时路上很少”的核心原因，在于代码里的一个致命距离阈值与名额限制发生了冲突。

在 `src/sceneEnemies.js` 的 `updatePrey` 中有这样一段代码处理猎物的淘汰（Culling）：

```javascript
const distanceFromCenter = Math.hypot(prey.x - this.player.centroidX, prey.y - this.player.centroidY);
if (distanceFromCenter > 3200) {
    this.finishPreyChaseTelemetry(prey, 'escaped', { reason: 'out-of-range' });
    // swap-and-pop 剔除
    // ...
}
```

**发生了什么？**
1. 每一关的猎物上限其实配得很小（例如在 `constants.js` 中，Stage 1 `forage-runner` desired 只有 7，`forage-school` 只有 6）。
2. 在平时的狩猎中，可能你会吓跑一些猎物，或者你并没有吃光它们，随着你向前移动，这些未能被猎杀的怪在屏幕后方的视野之外存活着。
3. 你的可见视野大概只在距离玩家几百到一千个单位的级别，**但这批猎物要直到离你 3200 距离外才会被 Despawn (删除)！**
4. 于是，你走过的路上产生了大量的 **“僵尸猎物”**（它们游荡在屏幕外，你看不见，但没有完全失效）。它们依然占据着 `alive` 的计数名额。
5. 因为名额一直大于等于 `rule.desired`，前方的刷新系统进入了长久的“静默期”，**此时你向前走就是一片荒芜的空地。**
6. 直到你的走位让某些落在身后的怪好不容易超过了 `3200` 从而死亡空出 1 个名额，这时候你的屏幕侧边又“刷”出一只来。这就直接暴露了“为了填数而刷怪”的程序味道。

---

## 3. 如何获得真正的“生物群落与散落生态感”？

如果要契合 README 里面提倡的“这不是机械编队而是有骨相的生物组织，世界是深邃的生态”，目前的产生机制急需一次体验升级。针对当前程序架构，给出两个层级的建议方案：

### 方案 A：快速手术级优化（利用现有体系产生“假区域感”）
在不改写数据结构的前提下，通过调整这套系统让它产生欺骗感：

1. **大幅削减剔除半径（Aggressive Culling）**
   - 把 `3200` 改为基于视野计算，比如 `const cullDistance = Math.max(1600, worldHalfWidth * 1.5);`。
   - 这意味着一旦你“路过”了某批猎物且它们没有向你产生仇恨，它们应该迅速在视距外释放内存和计数名额。
   
2. **重构日常刷新器的行为学（Pre-Spawn in Path）**
   - 不要让 `spawnConfiguredPrey` 永远在上下左右“四条边”均匀刷新。
   - **向玩家前进向量的前方区域投递**：读取 `player.intent.flowX/flowY`（也就是当前的运动趋势），在前方视线外的一个**宽阔环形区域（例如前方 1000-1800 处）**预先撒下一批猎物群体（使用类似开局的 `seed` 逻辑而不是单独一个个体插进来）。
   - 让玩家是通过“探索”走进了它们的驻地，主动发现了正在 Graze 或 Schooling 的群落，而不是“走着走着，屏幕边框吐出来一条鱼向你飘来”。

### 方案 B：中长期重构（迈向真正的世界块状渲染）
如果你想追求真正让人震撼的“开阔生态世界漫游感”，那么抛弃这套“街机刷怪法”。
1. **采用无缝 Chunk 种子机制 (Spatial Spawning Grid)**：
   - 用大网格（如 2000x2000 为一个 Chunk）管理世界。结合 Perlin Noise 或者简单的哈希种子，为每一个网格块在第一次靠近时赋予“该区域资源”（比如这是个“重兵把守群落”还是“小型游鱼散落区”）。
   - 只渲染/激活玩家所在 Chunk 及相邻 Chunk 的生物物理逻辑。
   - 当玩家离开较远 Chunk 时，冻结记录该块生态剩余，或直接销毁。
2. 这样做的最大好处是，猎物不再是被玩家拉扯着刷新的。这块地本身就有这些群落存在，玩家游走的不再是一块空白宣纸加上随机扔下的小怪，而是一块已经印有猎物驻点斑块的世界版图。

---

### 总结
你感觉怪，是因为**开场是真正意义上的环绕群落分布生成**，而**运行时退化为了街机式的填缝刷怪**。加之高达 `3200` 的内存淘汰距离，让你在大量日常时间里遇到了“怪物被卡在几条街以外，占据后台上限，导致前方空空荡荡”的程序悖论。

建议第一步直接去 `src/sceneEnemies.js` 调下 `updatePrey` 的那行 `if (distanceFromCenter > 3200)`，然后再稍微调整 `constants.js` 里平时生成时的 `packMin` / `packMax`，让前方一次性刷一小窝，而不是排队刷单只。这能立即解决 80% “头重脚轻”的空旷感。
