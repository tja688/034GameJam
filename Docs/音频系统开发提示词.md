
### 一、项目背景与总体目标

这是一个已经接近 game jam 尾声的 Phaser3 游戏项目。
目前程序结构是偏原生 JS / Mixin 式场景拆分，开发态里已经存在调参面板、JSON 写回、本地 dev server、编辑态等工具链。
旧文档不一定完全准确，你不要机械照搬，但要尊重现有项目的组织方式和开发节奏。

我要你做的不是“纯播放音频”，而是：

1. **先扫描代码里所有可能触发音频的地方**
2. **抽象出统一的 AudioManager / 音频事件层**
3. **设计一个开发态可用的音频调试面板**
4. **让面板能关联已接入的大音频素材库**
5. **让我能快速试听、切换、比较、应用、持久化**
6. **下次进游戏时，已经确认过的音频绑定和参数仍然保留**
7. **以后我新增内容时，也能按这套规则继续接音频，不再到处手写零散播放逻辑**

你要把这套东西做成一个**好用、便捷、快速**的音频调试 + 确定器，而不是一个半成品 demo。

### 四、系统总体架构要求

请按下面的思路实现：

#### 1. 先做 AudioManager

它应该成为全项目唯一可信的音频接入层。

至少负责：

* 音频资源注册
* 音频事件到素材的映射
* 播放、停止、淡入淡出
* 分类总线管理（至少 master / bgm / sfx / ui / ambience）
* 参数读取
* 开发态覆盖配置
* 持久化读写
* 调试试听 API
* 并发限制、冷却、去重等基础保护

#### 2. 建立“音频事件”层

不要允许业务代码到处直接写素材 key。
业务代码尽量写成这种形态：

* `audio.playEvent("player_pulse_tick")`
* `audio.playEvent("prey_devoured")`
* `audio.playEvent("ui_click")`

不要继续鼓励：

* 到处直接写文件名
* 到处硬编码 volume / rate
* 各模块各播各的

#### 3. 建立音频素材库清单

我要能接受“整个大资源文件夹都加进去”，你来考虑加载策略和进度条。
但实现上不要蠢到把所有东西无脑一次性卡死加载。

请设计一个适合当前项目的方案，比如：

* 音频库 manifest / index
* 分组与标签系统
* 懒加载 + 预加载混合
* 大资源接入时的进度反馈
* 开发态可容忍加载稍重，但不要把游戏体验搞崩

#### 4. 建立可持久化的音频配置

需要至少区分：

* 素材库定义
* 音频事件绑定
* 调试态临时覆盖
* 最终应用后的正式配置

我希望“点击应用”后保存下来的内容，下次进游戏依然有效。
你可以根据现有项目的 JSON 写回能力、调参面板写盘方式、本地 dev server 能力来选择最合适的落地方式。
优先考虑和当前项目已有的调参与写盘工作流兼容。

### 五、音频调试面板必须具备的能力

请做一个**轻量但高频实用**的面板，不追求花哨，追求顺手。

至少要有：

#### A. 事件浏览区

* 列出扫描得到的音频事件
* 支持搜索、分类、过滤
* 能看到事件来源模块 / 场景 / 类型
* 最好能显示“这个事件最近有没有触发过”

#### B. 当前事件配置区

* 当前绑定的素材 / 素材池
* 音量
* pitch / rate
* detune（如果适合）
* loop
* cooldown
* 并发上限
* 所属 bus / category
* 随机播放策略
* 是否启用

#### C. 素材库浏览区

* 浏览已接入音频库
* 支持搜索、标签、分组
* 快速试听单个素材
* 快速切换当前事件绑定
* 最好支持 A/B 对比或候选轮播试听

#### D. 试听与调试区

* 直接试听当前事件
* 停止当前试听
* 强制重播
* 能看到实际生效参数
* 最好能在游戏运行中即时生效，不强依赖刷新

#### E. 应用与持久化

* 应用到当前事件
* 应用后立刻更新运行时
* 保存到本地持久配置
* 可恢复默认
* 可导出 / 查看差异（如果实现成本不高）

如果可以，请尽量做到：

* **开发态覆盖与正式配置之间关系清晰**

### 八、接入策略要求

请先做音频管理器，再反向适配项目代码。

也就是说：

1. 先定义统一的 AudioManager 能力边界
2. 再定义统一的音频事件规范
3. 再去适配现有各模块
4. 最后把音频调试面板接到 AudioManager 上

不要反过来先在一堆地方乱塞播放代码。

目标是：
**管理器先立规矩，项目逻辑再按规矩接入。**

以后我新增功能，也能按同样规则加新的音频锚点和事件。
