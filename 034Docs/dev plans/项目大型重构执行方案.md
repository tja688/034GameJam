

## 0. 文件定位

本文是一次性大型结构重构的施工方案。

目标不是继续在现有工程上补丁式清理。
目标是把当前“可玩但混乱、耦合严重、调试负担高”的项目，重构为：

- 核心体验保留
- 正式模块可定位、可复用
- Playground 可独立测试
- 主流程可快速回到问题现场
- 调参工具不再是万能上帝面板
- 文档重建为唯一事实来源
- 后续 AI 修改可以低上下文、低波及面地进行

---

## 1. 重构结论（先看）

### 1.1 这次重构的本质
这次不是“继续分文件”。
这次是一次 **结构性脱钩重构**。

现有工程已经不是简单的原型，而是：
- 正式 demo 循环
- 五阶段 progression
- prey archetype
- HUD / world readability
- 主菜单 / 存档 / 调参 / 慢放编辑态
全部混在同一套运行结构里。

问题不在于它不能跑。
问题在于：
- Scene 承载过多系统职责
- 调试面板承担了太多开发中枢功能
- 功能分文件 != 依赖分层
- AI 后续修改的波及面太大
- 很难为“主角 vs 某 prey / 某姿态 / 某遭遇”建立稳定复现入口

### 1.2 这次重构的正确方向
这次重构必须做到：

1. 保留核心牵引移动体验
2. 把 Scene 从“万能神类”降级为“壳 + 入口”
3. 建立正式模块层
4. 建立独立 Playground
5. 建立 Scenario / Checkpoint / Encounter 验证入口体系
6. 拆分上帝调试面板
7. 废弃旧文档体系，重建唯一事实来源文档

## 2. 重构边界

### 2.1 必须保留的核心
以下内容视为“核心体验资产”，重构时保守处理：

- 意图场驱动，而非直接位移
- 脉冲巡游驱动节奏，而非单纯特效
- 节点抓地、前探、回弹、拖拽形成牵引感
- 节点角色差异（圆 / 方 / 三角）对运动与捕食有真实影响
- 收缩 / 舒张是有效决策，不是纯视觉动画
- 通过姿态去解决 prey，而不是平庸输出 check
- 世界内无文字可读性方向继续保留

### 2.2 可以激进重构的部分
以下内容允许彻底重做组织方式：

- CoreDemoScene 的职责拆分
- Mixin 组织方式
- 调参面板的结构与分工
- 编辑态的入口和工具组织
- 主流程入口组织方式
- 文档体系
- 开发态命令入口
- fixture / scenario / preset 数据组织方式

### 2.3 明确非目标
本次重构不做：

- 重型可视化编辑器
- 复杂拖拽式蓝图系统
- 每个对象一个 Scene
- 运行时代码级无缝热替换
- 大而全平台化资产管理系统
- 为了抽象漂亮而过度设计
- 推翻牵引移动核心重写玩法

## 4. 顶层运行结构重写

### 4.1 现状问题

当前入口是：

- `index.html`
    
- `src/main.js`
    
- `Object.assign(...)` 混入 `CoreDemoScene`
    
- `CoreDemoScene` 下再驱动输入 / movement / combat / enemies / render / progression / saveLoad
    

这种方式适合早期快速堆叠，但不适合后期模块化和 Playground。  
重构后必须把 Scene 降级为“壳”。

### 4.2 目标

Scene 只保留：

- 场景生命周期
    
- 模式切换
    
- overlay 切换
    
- runtime coordinator 驱动
    

所有实质逻辑下沉到 `core/runtime` 和 `modules/*`。

### 4.3 Runtime Coordinator

建立统一协调器：

```js
preUpdateInput()
updateIntent()
updateTopology()
updateMovement()
updatePrey()
updatePredation()
updateProgression()
updateCamera()
updateHud()
render()
```

由 coordinator 驱动，不再散落在 scene mixin 主循环内。

### 4.4 GameSessionState

建立统一会话状态容器：

```js
{
  player,
  activeNodes,
  links,
  prey,
  runState,
  cameraRig,
  uiState,
  debugState,
  sessionFlags
}
```

原先散落在 Scene 上的运行期状态，统一迁入这里。  
Scene 不再自己持有过多业务对象。

## 5. 正式模块层建立方案

### 5.1 第一批必须升格的模块

第一批正式模块必须包括：

- `playerCluster`
    
- `prey.skittish`
    
- `prey.school`
    
- `prey.bulwark`
    
- `prey.weakspot`
    
- `prey.apex`
    
- `hud.main`
    
- `camera.invasive`
    
- `progression.demo5stage`
    

### 5.2 模块标准

每个正式模块至少有：

```text
moduleName/
  *.module.js
  *.factory.js
  *.behavior.js
  *.config.js
  *.presets.js
  *.playground.js
```

### 5.3 模块职责

#### `*.module.js`

模块元信息：

- id
    
- category
    
- tags
    
- version
    
- factory 引用
    
- default config
    
- presets
    
- playground meta
    

#### `*.factory.js`

统一创建入口：

- 接收 scene / state / spawn request / preset / overrides
    
- 返回正式实例
    

#### `*.behavior.js`

该模块的运行逻辑。

#### `*.config.js`

默认参数与配置 schema。

#### `*.presets.js`

具名预设。

#### `*.playground.js`

Playground 展示信息：

- 标签
    
- 搜索词
    
- 默认 fixture
    
- 允许 inspector 暴露的字段
    

### 5.4 迁移原则

- 先迁 prey
    
- 再迁 playerCluster
    
- 再迁 HUD / camera / progression
    
- 迁移过程中允许通过 `legacy/migration-adapters/` 做临时桥接
    
- 桥接只允许短期存在，最终必须移除

## 6. Playground 重构方案

### 6.1 必须实现的目标

Playground 必须成为独立、可用、轻量的强测试空间，支持：

- 搜索资产
    
- 分类浏览
    
- 点击生成对象
    
- 生成玩家 / prey / 建筑 / 墙体 / 事件占位
    
- 组合测试
    
- 加载 fixture
    
- 保存 fixture
    
- 少量通用 inspector
    
- 从主流程进入
    
- 退出后返回主流程
    

### 6.2 架构

使用两个 Scene：

- `PlaygroundWorldScene`
    
- `PlaygroundOverlayScene`
    

#### `PlaygroundWorldScene`

负责：

- 世界模拟
    
- 模块生成
    
- fixture 运行
    
- 重置 / 清场
    

#### `PlaygroundOverlayScene`

负责：

- 搜索
    
- 分类浏览
    
- 轻量 inspector
    
- fixture 切换
    
- 返回主流程
    

### 6.3 Playground 与主流程解耦方式

主流程和 Playground 共享：

- `ModuleRegistry`
    
- `SpawnService`
    
- 正式模块实现
    
- 数据 schema（fixture / preset / scenario）
    

主流程和 Playground 不共享：

- 运行时实例对象
    
- 直接状态引用
    
- 运行态热替换逻辑
    

### 6.4 Inspector 范围

只允许 4 类字段：

- number
    
- boolean
    
- enum
    
- action
    

不允许快速膨胀为全量属性编辑器。

### 6.5 Fixture 结构

每个 fixture 都是数据，不是写死代码。

示例：

```json
{
  "id": "combat/bulwark-vs-player-compress",
  "spawn": [
    { "moduleId": "player.cluster", "preset": "debug-default", "x": 200, "y": 300 },
    { "moduleId": "prey.bulwark", "preset": "basic", "x": 520, "y": 300 }
  ],
  "camera": { "zoom": 1.0 },
  "notes": "验证收缩姿态能否稳定破 bulwark"
}
```

### 6.6 Playground 最小验收标准

满足以下全部条件才算 Playground MVP 完成：

1. 能搜索并生成 `player.cluster`
    
2. 能搜索并生成 5 类 prey
    
3. 能加载至少 6 个 fixture
    
4. 能保存 fixture
    
5. 能通过 inspector 改少量关键字段
    
6. 能从主流程进入并返回
    
7. 不依赖旧上帝面板才能完成基础测试

## 7. 主流程高颗粒度验证入口体系

### 7.1 目标

不用热替换，而是通过细粒度入口快速回到问题现场。

### 7.2 概念定义

#### Scenario

问题现场总入口。  
例：

- `pressure/objective/first-weakspot`
    
- `encircle/apex-first-pressure`
    

#### Checkpoint

进入该现场时玩家状态。  
包括：

- player preset
    
- energy
    
- growth cursor
    
- camera
    
- seed
    
- 必要状态标志
    

#### Encounter

当前现场生成什么。  
包括：

- prey 组合
    
- 地形
    
- 目标
    
- 特殊事件
    

### 7.3 数据结构样例

```json
{
  "id": "encircle/apex-first-pressure",
  "scene": "MainFlowScene",
  "checkpoint": {
    "playerPreset": "stage-encircle-mid",
    "energy": 0.72,
    "growthCursor": 3,
    "cameraZoom": 0.95
  },
  "encounter": {
    "spawn": [
      { "moduleId": "player.cluster", "preset": "stage-encircle-mid", "x": 220, "y": 300 },
      { "moduleId": "prey.apex", "preset": "objective-small", "x": 620, "y": 260 },
      { "moduleId": "prey.school", "preset": "support-pack", "x": 700, "y": 360 }
    ]
  }
}
```

### 7.4 开发态入口

必须支持以下任意一种进入方式：

- URL query
    
- dev command palette
    
- 最近使用 scenario 列表
    
- Playground “send to scenario”
    

### 7.5 最小验收标准

Scenario MVP 完成标准：

1. 至少有 8 个 scenario
    
2. 覆盖 5 个 stage 中至少各 1 个代表性现场
    
3. 至少有 2 个“主角 + 指定 prey + 指定姿态”的验证入口
    
4. 能一键跳入，不依赖完整重打流程

## 8. 上帝调试面板拆分方案

### 8.1 目标

把当前万能调参面板拆成多个职责清晰的系统。

### 8.2 拆分后结构

#### A. Baseline Tuning Panel

职责：

- 查看参数
    
- 搜索参数
    
- 改参数
    
- 差异导出
    
- 写入基线
    

不负责：

- 对象级测试入口
    
- 玩法场景入口
    
- fixture 运行
    

#### B. Playground Inspector

职责：

- 当前 fixture 内对象的小范围参数调试
    
- 快速查看对象状态
    
- 轻量调参
    

不负责：

- 仓库全量参数总控
    

#### C. Dev Command Layer

职责：

- 打开 Playground
    
- 打开某个 scenario
    
- 重载 fixture
    
- 切 debug overlay
    
- 返回主菜单 / 继续流程
    

#### D. Edit Tools

保留慢放、选中、拖点、删点、撤销，但它们不再与“大而全调参系统”绑死。

### 8.3 保留的能力

以下能力必须保留：

- 慢放
    
- 选择
    
- 拖点
    
- 删点
    
- 撤销
    

### 8.4 可删除 / 可迁出的能力

以下内容不应继续成为“调试面板中心职责”：

- 所有测试流程入口
    
- 所有局部对象验证
    
- 所有 fixture 组织
    
- 所有主流程回放入口

## 11. AI 在重构过程中的硬规则

### 11.1 AI 必须遵守

- 不允许一边重构一边继续扩玩法
    
- 不允许先做文档再做结构
    
- 不允许先做重型编辑器
    
- 不允许做热替换系统
    
- 不允许把 Playground 做成低配 Unity
    
- 不允许把 Scene 再次变回万能神类
    
- 不允许复制一份平行 prey / player 实现给 Playground 使用
    

### 11.2 AI 修改优先级

重构过程中优先操作：

- 新目录与新骨架
    
- runtime / state / registry / spawn
    
- prey 模块迁移
    
- playerCluster 模块迁移
    
- Playground MVP
    
- Scenario MVP
    

重构过程中谨慎操作：

- 核心 movement 力学公式
    
- pulse runner 节奏
    
- topology 关键结构规则
    
- predation 核心判定
    

### 11.3 AI 输出要求

每次完成一阶段，AI 必须说明：

1. 本阶段新建了哪些目录 / 文件
    
2. 哪些旧文件仍是桥接态
    
3. 哪些模块已完成迁移
    
4. 哪些能力仍依赖 legacy adapter
    
5. 当前是否已达到该阶段验收标准

## 12. 重构完成判定标准

### 12.1 架构完成

满足以下条件才算重构完成：

1. `CoreDemoScene` 不再是业务总控器
    
2. 运行时主逻辑由 `RuntimeCoordinator` 驱动
    
3. 5 个 prey archetype 都有正式模块目录
    
4. `playerCluster` 成为正式模块
    
5. Playground 可独立使用
    
6. 至少 8 个 scenario 可直接进入
    
7. 调参系统职责已拆开
    
8. 旧 README 已废弃
    
9. 034Docs\source已成为唯一事实来源
    

### 12.2 体验不退化

以下体验不能明显退化：

- 集群牵引移动感
    
- 收缩 / 铺开决策意义
    
- prey 逼姿态的成立性
    
- 无文字 HUD 的基本可读性
    
- 编辑态慢放 / 拖点 / 删点 / 撤销能力
    

---

## 13. 本次重构后的直接收益

完成后，项目必须获得这些实际收益：

- AI 改某个 prey 时，只需要定位到对应模块目录
    
- AI 改主角独立表现时，不再需要读完整个 demo scene 体系
    
- 你可以直接在 Playground 单测主角 / prey / 组合交互
    
- 你可以从 Scenario 快速回到问题现场
    
- 上帝面板不再承担全部开发职责
    
- 文档重新变成可信的唯一事实来源
    
- 后续开发进入“模块化迭代”而不是“继续整理屎山”
